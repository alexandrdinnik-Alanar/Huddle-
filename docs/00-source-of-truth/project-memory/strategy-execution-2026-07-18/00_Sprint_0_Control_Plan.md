Дата: 2026-07-18  
Статус: рабочий план  
Цель: перевести Huddle из состояния разрозненных материалов в управляемую систему реализации.

## Главный принцип

Сейчас мы не начинаем сразу “делать всё”: сайт, MVP, гранты, презентации, инвесторов и дизайн одновременно. Мы сначала собираем главный стратегический слой, из которого потом будут производиться все остальные документы и действия.

Правильная последовательность:

1. Huddle Master Project Bible.
2. Claims & Source Register v2.1.
3. Document Map.
4. Team & AI Agent Operating Model.
5. Написание Master Project Bible по разделам.
6. Полная визуальная презентация проекта.
7. Короткая публичная версия.
8. Эмоциональная сжатая версия.
9. Funding / investor / municipality / B2B packages.
10. Техническое задание.
11. Дизайн-система, промо-сайт и demo.

## Почему именно так

Если сразу делать сайт или презентации, мы снова получим красивые, но несогласованные материалы. Huddle большой: family ecosystem, marketplace, activities, experts, municipalities, public value, funding, privacy, safety, business model. Поэтому сначала нужен единый “мозг проекта” — Master Project Bible.

## Gates

### Gate A — Strategic Coherence

Материал проходит, если он не противоречит:

- Oslo-first launch;
- evidence-before-scale;
- family trust and child safety;
- финальному бренду Huddle;
- статусу pre-validation.

### Gate B — Evidence

Любая цифра должна иметь один из статусов:

- Verified source;
- Owner confirmed;
- Planning assumption;
- Hypothesis;
- Needs verification;
- Not approved for external use.

### Gate C — External Readiness

Документ можно отправлять наружу только если:

- убраны внутренние комментарии;
- все claims проверены;
- финансовые данные согласованы;
- legal/applicant wording верен;
- прошёл founder approval.

## Current risks

| Риск | Почему важно | Контроль |
|---|---|---|
| Снова смешать брендбук, сайт и продукт | Это разные артефакты | Отдельные document tracks |
| Отправить документы без источников | Потеря доверия | Claims Register обязателен |
| Говорить о гипотезах как о доказанных фактах | Риск для инвесторов и грантов | Evidence labels |
| Начать слишком большой MVP | Scope overload | MVP open decisions |
| Делать дизайн до фиксации структуры продукта | Потеря времени | Design starts from approved scope |

## Sprint 0 definition of done

Sprint 0 считается закрытым, когда:

- структура Master Project Bible утверждена;
- Claims Register v2.1 создан и содержит первый список ключевых утверждений;
- Document Map создан и разделяет все документы по аудиториям;
- Team & AI Agent Operating Model создан;
- понятен следующий раздел Master Project Bible, который пишем первым.

Текущий статус после продолжения работы:

- Master Project Bible v0.1 создан по разделам 1–25;
- Master Project Bible v0.2 собран в один документ;
- первый QA-pass выполнен;
- следующий шаг: claim-ID pass и founder voice для Section 2.
