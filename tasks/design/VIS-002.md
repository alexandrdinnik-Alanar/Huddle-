# VIS-002 — Create the canonical Figma production foundation

## Task ID

`VIS-002`

## Objective

Create Huddle's first real production design foundation in Figma. No approved route-level product design currently exists.

This task establishes the canonical Figma file, governed design system, and approved layout frames needed before React base layouts can be implemented in S0-015.

## Dependencies

- `S0-014` merged.
- locked brand system available in `docs/00-source-of-truth/brand-system.md`;
- approved runtime brand assets available under `public/brand/`;
- design references available under `assets/screens/` and `assets/reference/claude-design-2026-07/`;
- product owner confirms the file and frames created by this task.

## Source precedence

1. product-owner decisions recorded during VIS-002;
2. locked repository brand system and approved assets;
3. accessibility requirements and semantic design tokens;
4. canonical route map and product priorities;
5. PNG and Claude Design reference corpora;
6. historical or archived Figma files.

Reference material may inform the work but does not authorize copying or production status.

## Required Figma file structure

Create or formally designate one canonical Huddle production design file with these minimum pages:

- `00_Cover & Governance`
- `01_Brand Foundation`
- `02_Variables & Styles`
- `03_Components`
- `04_Public Layouts`
- `05_Auth Layouts`
- `06_App Layouts`
- `07_Admin Layouts`
- `08_Approved Screens`
- `09_Prototype & Flows`
- `99_Archive`

Page names may be refined during execution, but the final names must be recorded in the handoff and repository registries.

## Required design foundation

Create governed Figma variables, styles, or equivalent reusable definitions for:

- locked brand colors;
- accessible text and action colors;
- surface and border semantics;
- typography families, sizes, weights, and line heights;
- spacing scale;
- radius scale;
- shadows/elevation;
- responsive breakpoints and frame widths;
- focus, disabled, error, success, warning, and information states.

Do not introduce a competing brand palette or unapproved logo treatment.

## Required reusable components

At minimum:

- approved Huddle Family logo placement component;
- Public header and footer;
- Auth brand header or lockup block;
- App top navigation and mobile bottom navigation;
- Admin sidebar and header;
- buttons and links used by the layouts;
- structural container, section, and surface primitives;
- accessibility-ready focus states.

Components must use Auto Layout and reusable variables/styles where supported.

## Required layout frames

Create desktop and mobile frames for:

### Public

- public header;
- public content shell;
- public footer;
- navigation open/closed states where applicable.

### Auth

- centered or otherwise approved auth shell;
- logo/brand placement;
- content card or content region;
- privacy and trust note region.

### App

- authenticated app shell;
- top navigation;
- mobile bottom navigation;
- content region;
- safety/privacy access point;
- empty structural state without product features.

### Admin

- admin shell;
- sidebar;
- header;
- content region;
- audit/security notice region;
- empty structural state without admin feature controls.

## Approval requirements

Before completion:

1. product owner reviews the file structure;
2. product owner reviews desktop and mobile frames for all four surfaces;
3. requested corrections are applied;
4. product owner explicitly approves the final frames;
5. stable page names and exact node IDs are recorded in the repository;
6. only approved layout records receive `PRODUCTION` status.

## Repository updates required after approval

- `docs/00-source-of-truth/design-source-of-truth.md`
- `docs/01-product/design-source-inventory.csv`
- `docs/01-product/screen-route-registry.csv` or a dedicated layout registry if route-level mapping is not yet appropriate
- `docs/execution/VIS-002/HANDOFF.md`
- `NEXT_TASK.md`, changing the next task to S0-015 only after approval

## Out of scope

- React implementation;
- product feature screens;
- marketplace, activities, events, community, messaging, finance, municipality, or moderation flows;
- full 105-route design coverage;
- copying raw Claude HTML into Figma;
- marking unapproved reference screens as production;
- creating fake data-backed product dashboards.

## Acceptance criteria

- one canonical Figma production file is designated;
- required pages exist and are organized;
- variables/styles cover the required foundation;
- required reusable shell components exist;
- Public, Auth, App, and Admin desktop/mobile layout frames exist;
- frames use the locked brand system;
- accessibility and responsive behavior are documented;
- product-owner approval is recorded;
- exact page names and node IDs are committed to the repository;
- no React or product feature implementation is included.

## Stop condition

Stop after VIS-002 is approved and documented. Do not begin S0-015 in the same task or branch.
