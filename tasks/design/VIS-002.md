# VIS-002 — Create the Huddle production design foundation

## Task ID

`VIS-002`

## Objective

Create Huddle's first real production design foundation. No approved route-level product design currently exists.

The approved workflow is **VS Code-first, Figma-governed and component-system driven**:

1. explore distinct art directions as isolated coded concepts;
2. review the concepts in a real responsive browser environment;
3. select and refine one visual territory;
4. establish the reusable component and token system;
5. transfer the approved system and layout frames into the canonical Figma file;
6. record approval and stable Figma node IDs before S0-015 begins.

Claude Design HTML and repository PNGs are structural and UX references only. They are not finished visual design.

## Dependencies

- `S0-014` merged;
- locked brand system available in `docs/00-source-of-truth/brand-system.md`;
- approved runtime brand assets available under `public/brand/`;
- design references available under `assets/screens/` and `assets/reference/claude-design-2026-07/`;
- 2026 art-direction research recorded in `docs/01-product/huddle-art-direction-research-2026.md`;
- product owner reviews and approves the final direction.

## Source precedence

1. product-owner decisions recorded during VIS-002;
2. locked repository brand system and approved assets;
3. accessibility, responsive and performance requirements;
4. canonical route map and product priorities;
5. coded Design Lab evidence from real browser layouts;
6. PNG and Claude Design reference corpora;
7. historical or archived Figma files.

Reference material may inform the work but does not authorize copying or production status.

## Stage D1 — Coded art-direction concepts

Create an isolated non-production route at `/design-lab` containing three materially different directions:

- `A — Nordic Editorial`;
- `B — Local Discovery Atlas`;
- `C — Human Community Journal`.

Each direction must demonstrate the same comparison set:

- navigation;
- homepage hero;
- discovery/search composition;
- activity card or activity-list language;
- trust/safety signal;
- desktop and mobile responsive behavior.

The concepts must differ in composition, typography, information rhythm, surface treatment and interaction language — not merely color.

### D1 rules

- `/design-lab` is internal, `noindex` and not linked from production navigation;
- fictional demonstration content must be clearly identified;
- do not modify `/`, `/activities`, auth, app or admin production routes;
- do not add data fetching, authentication, product APIs or fake dashboards;
- do not add a generic UI theme or copy a template;
- do not add a production dependency unless separately approved;
- CSS experimentation may be scoped to the Design Lab, but production tokens remain unchanged;
- support keyboard focus, readable contrast, mobile layouts and reduced motion;
- the product owner must compare all three concepts before one is promoted.

## Stage D2 — Direction selection and foundation

After product-owner review:

- record the selected direction and rejected alternatives;
- refine typography, palette, spacing, radius, borders, elevation, imagery and motion principles;
- document accessibility and performance constraints;
- define which experimental values become canonical tokens;
- remove visual ideas that do not scale to product surfaces.

No direction is approved merely because it exists in `/design-lab`.

## Stage D3 — Component system

Create and review the minimum reusable system needed for layout work:

- logo placement;
- buttons and links;
- fields and search controls;
- chips and filters;
- structural containers and sections;
- Public header and footer;
- Auth brand region;
- App top navigation and mobile bottom navigation;
- Admin sidebar and header;
- focus, disabled, error, success, warning and information states.

Storybook may be introduced only through a separate dependency decision and task-scoped change.

## Stage D4 — Required layout frames

Create desktop and mobile designs for:

### Public

- public header;
- public content shell;
- public footer;
- navigation open/closed states where applicable.

### Auth

- auth shell;
- logo/brand placement;
- content region;
- privacy and trust note region.

### App

- authenticated app shell;
- top navigation;
- mobile bottom navigation;
- content region;
- safety/privacy access point;
- empty structural state without product features.

### Admin

- admin shell;
- sidebar;
- header;
- content region;
- audit/security notice region;
- empty structural state without fake controls.

## Stage D5 — Figma governance

Create or formally designate one canonical Huddle production design file with these minimum pages:

- `00_Cover & Governance`
- `01_Brand Foundation`
- `02_Variables & Styles`
- `03_Components`
- `04_Public Layouts`
- `05_Auth Layouts`
- `06_App Layouts`
- `07_Admin Layouts`
- `08_Approved Screens`
- `09_Prototype & Flows`
- `99_Archive`

Transfer the approved direction, tokens, components and layout frames into Figma. Page names may be refined, but final names and stable node IDs must be recorded.

## Approval requirements

Before VIS-002 completes:

1. product owner reviews all three D1 concepts;
2. one direction or an explicitly documented hybrid is selected;
3. requested corrections are applied;
4. desktop and mobile layouts for all four surfaces are reviewed;
5. the production design system is recorded in Figma;
6. product owner explicitly approves the final frames;
7. stable page names and exact node IDs are recorded in the repository;
8. only approved records receive `PRODUCTION` status.

## Repository updates required after approval

- `docs/00-source-of-truth/design-source-of-truth.md`
- `docs/01-product/design-source-inventory.csv`
- `docs/01-product/screen-route-registry.csv` or a dedicated layout registry
- `docs/execution/VIS-002/HANDOFF.md`
- `NEXT_TASK.md`, changing the next task to S0-015 only after final approval

## Out of scope

- production React layout implementation;
- product feature screens beyond concept fragments needed for art-direction comparison;
- marketplace, community, messaging, finance, municipality or moderation implementation;
- full 105-route design coverage;
- copying raw Claude HTML into React or Figma;
- marking unapproved concept screens as production;
- data-backed dashboards or fake admin controls.

## Acceptance criteria

- three materially different responsive coded concepts are available at `/design-lab`;
- the route is isolated and `noindex`;
- no production route or feature behavior is changed;
- the product owner selects and approves a visual direction;
- canonical tokens and reusable component principles are documented;
- Public, Auth, App and Admin desktop/mobile layout designs exist;
- one canonical Figma production file is designated and organized;
- approved variables, components and layout frames exist in Figma;
- accessibility and responsive behavior are documented;
- exact page names and node IDs are committed;
- no production feature implementation is included.
