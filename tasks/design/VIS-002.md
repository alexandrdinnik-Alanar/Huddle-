# VIS-002 — Reconcile and productionize the complete Huddle design corpus

## Task ID

`VIS-002`

## Objective

Continue from the completed Claude design inventory and route canonicalization. Reconcile the supplemental `Site.zip` screens into the existing corpus, preserve the established Huddle visual direction, and convert the combined reference system into governed, editable production design assets.

This task must not restart the old audit, reduce the project to the newest 100 screens, or invent an unrelated visual direction.

## Verified source baseline

The existing audited corpus contains:

- 315 HTML prototype records;
- 234 unique HTML contents after exact deduplication;
- 205 PNG files;
- 12 PDF files;
- 542 total files;
- 105 canonical routes;
- 286 route-screen/state records already mapped to route candidates.

The supplemental `Site.zip` contains:

- 100 numbered screens across 10 sets;
- 19 byte-identical overlaps with the earlier corpus;
- 24 visual variants of existing-route patterns;
- 52 new states on existing routes;
- one direct candidate for the previously uncovered `/app/seller/payouts` route;
- two route-embedded component references;
- six route candidates requiring product-owner decisions.

## Source precedence

1. product-owner decisions recorded during VIS-002;
2. the combined audited Huddle design corpus and its established cross-platform visual language;
3. locked repository brand assets and brand rules;
4. accessibility requirements and semantic design tokens;
5. canonical route map and product priorities;
6. approved reconstructed Figma components and frames;
7. merged React implementation.

No single ZIP, PNG set, HTML file, screenshot, Figma draft, or coded experiment may replace the combined corpus or silently redefine the style.

## Established visual direction

Preserve and professionally refine the existing Huddle platform identity:

- warm cream and off-white surfaces;
- dark forest-green navigation and primary actions;
- copper/orange Huddle accents;
- natural family and community photography;
- restrained rounded geometry;
- thin neutral borders and low-elevation surfaces;
- compact structured product layouts;
- consistent public, family, organizer, partner, municipality, community, service, payment and administrative patterns.

The superseded A/B/C art-direction experiments from PRs #34 and #35 are not design authority and must remain discarded.

## Required sequence

### 1. Preserve the completed baseline

Use these existing inventories as the starting point:

- `docs/01-product/claude-prototype-inventory.csv`;
- `docs/01-product/claude-route-canonicalization.csv`;
- `docs/02-architecture/canonical-route-map.csv`.

Do not recreate the 315-row prototype inventory from scratch.

### 2. Reconcile the supplemental package

For every `Site.zip` screen, record:

- source set and screen number;
- source filename and SHA-256;
- inferred screen name;
- product area and primary role;
- canonical route candidate;
- exact-overlap evidence;
- matched Claude screen candidates;
- relation to the existing corpus;
- reconciliation decision.

Allowed relation values:

- `EXACT_OVERLAP`;
- `EXISTING_ROUTE_VISUAL_VARIANT`;
- `EXISTING_ROUTE_NEW_STATE`;
- `CANONICAL_GAP_FILLED`;
- `COMPONENT_REFERENCE`;
- `NEW_ROUTE_CANDIDATE`.

### 3. Resolve route decisions

The following supplemental route candidates require explicit product-owner decisions before entering the canonical route map:

- `/community`;
- `/municipalities`;
- `/organizers`;
- `/organizers/[organizerId]`;
- `/partners`;
- `/app/resources`.

Do not add, merge or remove these routes silently.

### 4. Extract the governed design system

Derive one reusable system from the complete corpus:

- semantic color roles;
- typography hierarchy;
- spacing and responsive grids;
- radii, borders and elevations;
- public and authenticated navigation;
- sidebars and workspace headers;
- buttons, fields, selectors and filters;
- cards, lists, tables and dashboards;
- status, empty, loading, error and success states;
- dialogs, drawers, menus and mobile navigation;
- photography, icons and media rules;
- privacy, safety and accessibility patterns.

Resolve inconsistencies explicitly instead of selecting arbitrary variants.

### 5. Validate in VS Code and Storybook

Create coded responsive validation for representative shared patterns and shells. This work must preserve the existing style and may not become a parallel visual system.

Representative coverage must include:

- Public/discovery;
- Family app;
- Organizer workspace;
- Partner workspace;
- Municipality workspace;
- Community and messaging;
- Services and bookings;
- Wallet and payments;
- Admin/trust patterns where source coverage exists.

## Acceptance criteria

- the previous 315-row inventory and 105-route map are preserved;
- all 100 supplemental screens have traceable delta records;
- exact overlaps are retained without inflating canonical screen counts;
- `/app/seller/payouts` is recorded as a newly covered canonical gap;
- six route candidates are surfaced for product-owner decisions;
- one consolidated route-coverage map exists;
- one component and token inventory is derived from the complete corpus;
- representative responsive coded validation preserves the established style;
- the canonical Figma file contains editable governed components and approved representative frames;
- product-owner approval and exact node IDs are recorded before any record is marked `PRODUCTION`.

## Out of scope

- inventing a replacement art direction;
- treating `Site.zip` as the full corpus;
- counting every source file or state as a unique production page;
- copying raw HTML directly into production React;
- marking screenshots or unapproved Figma drafts as production;
- implementing S0-015 before VIS-002 approval.

## Stop condition

Stop after the full corpus is reconciled, representative responsive components and frames are approved, and the repository records the approved Figma authority. Do not begin S0-015 in the same task or branch.
