## 10. Required auth helper scope

Create repository-equivalent auth helpers under:

- `src/lib/auth/client.ts`
- `src/lib/auth/server.ts`
- `src/lib/auth/session.ts`
- `src/lib/auth/current-user.ts`
- `src/lib/auth/types.ts`

Create these required functions:

- `getSession()`
- `getCurrentUser()`
- `requireUser()`
- `requireActiveUser()`

## 11. Required client helper

`client.ts` must provide:

- a browser-safe Supabase client only
- use of public client environment variables only
- no service-role key
- no server-only secrets
- no privileged operations

## 12. Required server helper

`server.ts` must provide:

- a server-only Supabase access pattern
- alignment with current Next.js App Router conventions
- no leakage of server secrets into client bundles
- no custom session system

## 13. Required session helper

`session.ts` must:

- expose `getSession()`
- use Supabase Auth session state
- avoid custom cookies beyond framework-supported and Supabase-supported patterns
- avoid custom token minting
- avoid password handling

## 14. Required current-user helper

`current-user.ts` must:

- map Supabase-authenticated identity to application `User` data
- resolve roles through `UserRole` and `Role`
- return only safe app-user data
- handle missing app `User` records explicitly
- handle suspended and deleted user status explicitly within contract boundaries
- avoid implementing route guarding

## 15. Required auth types

`types.ts` must define the repository auth types needed for:

- auth-session access
- safe app-user data shape
- resolved active roles

Do not silently invent unrelated product types.

## 16. Required getSession behavior

`getSession()` must:

- return Supabase-authenticated session state
- fail explicitly when session retrieval fails
- avoid custom token formats
- avoid custom persistence
- remain compatible with server-side usage

## 17. Required getCurrentUser behavior

`getCurrentUser()` must:

- derive app-user state from authenticated Supabase identity plus application database records
- return only the approved app-user shape
- resolve roles from active `UserRole` rows only
- exclude revoked role assignments
- handle missing application user records explicitly
- avoid permission evaluation

Important integration ambiguity:

- the required output shape includes `onboardingCompleted`
- the current merged `User` schema in `prisma/schema.prisma` does not contain `onboardingCompleted`
- implementation must treat this as a factual blocker or ambiguity to be resolved from authoritative source before inventing storage or silently expanding schema
- `S0-009` must not silently add this field to `prisma/schema.prisma` without explicit contract correction

## 18. Required requireUser behavior

`requireUser()` must:

- require an authenticated user
- fail explicitly when unauthenticated
- avoid permission checks
- avoid role guards

## 19. Required requireActiveUser behavior

`requireActiveUser()` must:

- require an authenticated application user
- reject inactive user states according to `UserStatus` semantics
- avoid permission checks
- avoid role guards

## 20. Role-resolution requirements

- use existing `User` / `UserRole` / `Role` data
- return app-safe role values
- support multiple roles
- revoked role assignments must not be treated as active roles
- do not implement permission evaluation
- do not implement `RolePermission`
- do not implement policy engine

## 21. App-user shape

`getCurrentUser()` must return exactly the required safe shape:

- `id`
- `email`
- `name`
- `status`
- `roles`
- `onboardingCompleted`

Do not silently invent extra fields in the contract as required output.

## 22. Environment variable requirements

Required placeholders:

- `NEXT_PUBLIC_SUPABASE_URL`
- `NEXT_PUBLIC_SUPABASE_ANON_KEY`

Optional only if factually required server-side:

- `SUPABASE_SERVICE_ROLE_KEY`

Rules:

- placeholder values only
- no real secrets
- no production credentials
- no real endpoints
- no hardcoded credentials

## 23. Service-role-key policy

- do not add `SUPABASE_SERVICE_ROLE_KEY` by default
- only add it if implementation genuinely requires privileged server-side Supabase operations
- if added:
  - it must remain server-only
  - it must never be imported by client code
  - it must never be exposed through `NEXT_PUBLIC_*`
  - the requirement must be documented in the handoff
- prefer not using a service-role key for `S0-009` if a normal authenticated server client is sufficient

## 24. Security requirements

- no secrets committed
- no custom password storage
- no custom session database
- no custom token format
- no custom JWT signing
- no auth bypass
- no privilege escalation
- no client exposure of server secrets
- no trust in client-provided roles
- roles must come from the application database
- revoked role assignments must be excluded from active role resolution

## 25. Privacy requirements

- return only the required app-user fields
- no unnecessary profile data
- no raw auth-provider metadata unless explicitly required
- no secret or session-token logging
- no PII logging
- no real user seed data

## 26. Server/client boundary

- `client.ts` must be safe for browser use
- `server.ts` must remain server-only
- service-role secrets must never cross into client code
- current-user database resolution must happen server-side
- do not create cross-boundary imports that bundle server code into client code

## 27. Error-handling requirements

- authentication failures must be explicit
- missing app-user resolution must be explicit
- inactive-user rejection must be explicit
- do not leak secrets or raw provider tokens in errors
- do not hide factual blockers behind fake fallback user data
