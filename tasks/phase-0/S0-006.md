# S0-006 - Configure PostgreSQL and Prisma

## 1. Task ID

`S0-006`

## 2. Title

`Configure PostgreSQL and Prisma`

## 3. Objective

Configure the Huddle database foundation using PostgreSQL and Prisma without implementing any product feature models or downstream application features.

## 4. Dependency

- `S0-005` merged

## 5. Source precedence

Apply source precedence:

1. accepted architecture decisions and repository architecture contract;
2. merged repository implementation through `S0-005`;
3. Sprint 0 source pack;
4. roadmap ordering and quality gates.

For `S0-006`, `ADR-003` in `docs/02-architecture/adr-register.csv` is binding: PostgreSQL + Prisma with Supabase Postgres as the initial hosting target.

## 6. Allowed files

- `package.json`
- `pnpm-lock.yaml`
- `.env.example`
- `prisma/**`
- `src/lib/db/**`
- `tasks/phase-0/S0-006.md`
- `NEXT_TASK.md`
- `docs/execution/S0-006/**`

Additional files are forbidden unless strictly required and explicitly justified in the handoff. Do not modify `tsconfig.json` or `README.md` unless the change is unavoidable to complete the database foundation and the reason is written in the handoff.

## 7. Required implementation scope

Implement only the database foundation required by the roadmap and Sprint 0 source:

- PostgreSQL datasource configuration
- `DATABASE_URL` repository placeholder usage
- Prisma client generator configuration
- `prisma/schema.prisma`
- database client helper under `src/lib/db/**`
- package scripts:
  - `db:generate`
  - `db:migrate`
  - `db:push` only if justified for local development
  - `db:seed`
  - `db:studio`
- Prisma dependency and configuration required for the repository

This task establishes the foundation only. Product feature models belong to later tasks.

## 8. Required deliverables

- Prisma installed and configured for the repository
- `prisma/schema.prisma`
- PostgreSQL datasource using `DATABASE_URL`
- Prisma client generator configured
- database helper under `src/lib/db/**`
- required package scripts present
- `.env.example` includes `DATABASE_URL` placeholder
- `docs/execution/S0-006/HANDOFF.md`

## 9. Database configuration requirements

- database must be PostgreSQL
- datasource must use `DATABASE_URL`
- configuration must remain portable PostgreSQL as required by architecture
- Supabase Postgres is the initial hosting target, but repository configuration must not hardcode vendor-specific production assumptions

## 10. Prisma requirements

- install and configure Prisma
- create `prisma/schema.prisma`
- configure Prisma client generator
- ensure Prisma client generation can run successfully
- do not add product feature models in this task

## 11. Environment variable requirements

- `.env.example` must include `DATABASE_URL`
- use a placeholder value only
- no real credentials
- no production secrets
- no hardcoded database connection strings in source files

## 12. Package script requirements

Add or verify these scripts:

- `db:generate`
- `db:migrate`
- `db:push` only if justified
- `db:seed`
- `db:studio`

Scripts must be repository-standard and usable through `pnpm`.

## 13. Database client helper requirements

- create the database client helper under `src/lib/db/**`
- helper belongs to cross-cutting infrastructure, not product domain logic
- helper must not introduce product rules, feature queries or business workflows
- helper must support later Prisma-based tasks without creating feature-specific coupling

## 14. Forbidden scope

- product feature models
- marketplace models
- event models
- family models
- admin models
- moderation/report models
- auth implementation
- API feature work
- UI work
- route changes
- product screens
- seed data containing real personal data
- production secrets
- hardcoded database credentials

Do not begin `S0-007` identity models in this task.

## 15. Security requirements

- never commit real database credentials
- keep `DATABASE_URL` as placeholder-only in `.env.example`
- do not run any production-targeted database command
- no real personal data in seed or setup material
- no secrets in code, docs or scripts

## 16. Migration policy

- no production migration execution
- no destructive migration
- no schema reset
- no `db push` against production
- migrations must be explicit and reviewable
- `S0-006` establishes foundation only; product/identity schema evolution belongs to later approved tasks

## 17. Quality gates

- `python scripts/verify-brand-assets.py`
- `pnpm lint`
- `pnpm typecheck`
- `pnpm test`
- `pnpm build`
- `pnpm db:generate` or equivalent Prisma client generation validation
- no feature code introduced
