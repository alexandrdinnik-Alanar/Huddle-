# S0-014 — Create AuditLog model and audit helper

## Task ID

`S0-014`

## Objective

Create the append-only audit foundation required by future admin, moderation, privacy, role, and security-sensitive operations without implementing those product workflows.

## Dependency

`S0-013` is merged. The task may use the existing Prisma client foundation but must not create a product API.

## Source precedence

1. accepted ADRs and repository security rules;
2. actual merged repository implementation through S0-013;
3. Sprint 0 S0-014 source where not superseded.

## Required model

Add `AuditLog` with:

- `id` string primary key with generated default;
- optional `actorId`;
- optional historical `actorRole`;
- required `action`;
- required `targetType`;
- optional `targetId`;
- optional JSON `metadata`;
- optional `ipAddress`;
- optional `userAgent`;
- generated `createdAt`.

Add indexes for `actorId`, `action`, `targetType + targetId`, and `createdAt`.

An optional actor relation may use `ON DELETE SET NULL`. System/background events must work without a user actor.

## Required helper

Create `createAuditLog({ actorId, actorRole, action, targetType, targetId, metadata, requestContext })` under `src/lib/audit/**`.

The helper must:

- default actor-less events to role `SYSTEM`;
- normalize bounded identifiers and request context;
- discard invalid IP values;
- remove sensitive metadata keys recursively;
- bound metadata depth, key count, array size, and string length;
- return a typed result;
- not throw on write failure by default;
- support explicit throw-on-error behavior for workflows that require audit persistence to be transactional or fail-closed;
- never return or log the raw database error by default.

## Privacy and security

Never store raw:

- passwords, tokens, secrets, authorization headers, or cookies;
- message or free-text bodies;
- email addresses or phone numbers in metadata;
- exact address or precise location;
- child name, birthday, school, identifier, or other sensitive child data;
- complete request bodies.

Prefer stable IDs, event types, status codes, counts, scopes, versions, and hashes. Schema validation and authorization remain separate concerns.

## Append-only policy

S0-014 exposes create behavior only. No update or delete helper is allowed. Database-role enforcement, retention jobs, export controls, and audit viewer permissions are follow-up work.
