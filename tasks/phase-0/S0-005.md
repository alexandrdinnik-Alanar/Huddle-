## Required implementation

Create primitive components in:

```text
src/components/ui/
```

Required primitive components:

- `Button`
- `Input`
- `Textarea`
- `Select`
- `Checkbox`
- `Switch`
- `Modal`
- `Toast`
- `Badge`
- `Card`
- `Tabs`
- `Skeleton`
- `EmptyState`
- `ErrorState`

## Required variants and states

Button:

- variants: `primary`, `secondary`, `ghost`, `danger`
- sizes: `sm`, `md`, `lg`
- states: default, hover, disabled, loading, focus-visible

Badge:

- variants: `neutral`, `success`, `warning`, `danger`, `info`

Card:

- variants: `default`, `elevated`, `soft`

Form controls:

- support disabled state where applicable
- support error state for `Input`, `Textarea` and `Select`
- support focus-visible states

Feedback and utility primitives:

- `Skeleton` must support loading-placeholder use
- `EmptyState` and `ErrorState` must exist as generic reusable primitives
- `Modal` and `Toast` must remain primitive and internal, without feature-specific workflows

## Accessibility requirements

- preserve keyboard accessibility for all interactive components
- support meaningful labels and ARIA semantics where appropriate
- keep focus-visible styles explicit and token-driven
- ensure disabled states are communicated visually and semantically
- decorative-only elements must not be announced unnecessarily
- avoid interaction patterns that require pointer-only use

## Forbidden scope

- product screens or route page implementation
- marketplace, events, community, admin or municipality feature logic
- feature-specific cards such as `ListingCard` or `EventCard`
- app shell or layout work reserved for later tasks
- dark mode
- theme editor or user-selectable themes
- external UI kit adoption unless separately approved
- unrelated runtime, auth, database, API or route work
- replacing or bypassing the approved S0-004 design token foundation
