## 8. Material source conflict

The original Sprint 0 S0-011 source assumes `requireActiveUser`, resolved application roles from current-user/auth resolution, and `onboardingCompleted`.

Accepted S0-009 architecture explicitly deferred or removed those assumptions. Therefore, the original S0-011 cannot be implemented verbatim.

S0-011 must not invent app-user reconciliation, role loading, active-user resolution, onboarding state, canonical app `User` semantics from `SupabaseAuthenticatedIdentity`, provider subject id to domain `User.id` equivalence, or email-based canonical reconciliation.

P1-001 remains deferred. P1-005 remains deferred.

## 9. Accepted S0-009 architecture boundary

`getCurrentUser()` and `requireUser()` provide authenticated Supabase identity only.

The authenticated identity shape is:

```ts
{
  id: string;
  email: string | null;
}
```

S0-009 does not provide canonical application `User` reconciliation, resolved application roles, active-user status, or onboarding completion.

## 10. Accepted S0-010 permission boundary

Permission helpers are pure evaluators.

Permission-helper input is an explicit trusted role-bearing subject. The minimal shape is:

```ts
{ roles: readonly RoleKey[] }
```

S0-010 does not authenticate, load roles, query the database, reconcile identities, or guard routes.

## 11. Corrected S0-011 scope

Implement a route-guard library that composes existing S0-009 authentication helpers only for authentication-only semantics and composes S0-010 permission helpers only when an explicit trusted role-bearing subject is supplied by the caller.

Implement documentation and tests that make unresolved runtime role source, onboarding state, and active-user status gaps explicit.

## 12. Deferred original-source requirements

The following original-source assumptions are deferred:

- runtime active-user enforcement
- current-user/application-user reconciliation
- runtime role loading
- onboarding completion checks
- treating `SupabaseAuthenticatedIdentity` as canonical app `User`
- treating provider subject id as domain `User.id`
- email-based canonical reconciliation

## 17. Authentication guard semantics

`requireAuth` must be server-side and compose existing S0-009 `requireUser()`.

`requireAuth` proves authentication only. It does not prove canonical app `User`, roles, active status, or onboarding completion.

## 18. Explicit role-bearing subject semantics

Role-bearing guards must accept an explicit trusted subject from the caller.

Roles must use `RoleKey` values only. Do not accept arbitrary strings. Do not load roles.

## 19. Permission guard semantics

`requirePermission` must compose the S0-010 pure permission helper boundary.

It must accept an explicit trusted `PermissionSubject`, perform no auth lookup, perform no database lookup, and perform no role loading.

## 20. Ownership guard semantics

`requireOwner` is conditionally implementable only as a pure ownership comparison over explicit trusted identifiers.

It must not query the database, load a resource, compare Supabase identity id with domain `ownerId` as canonical ownership, or assume provider subject id equals domain `User.id`.

If a safe generic signature cannot be justified during implementation, defer `requireOwner` rather than inventing unsafe ownership semantics.

## 21. Status guard semantics

`status.ts` must not load active-user state, query the database, or reconcile app users.

It may contain only pure assertions over trusted pre-resolved status input or a narrow deferred placeholder. Runtime active-user enforcement remains deferred.

## 22. Moderator guard semantics

`requireModerator` must accept only these roles:

- `MODERATOR`
- `ADMIN`
- `SUPER_ADMIN`

No lower roles are accepted.

## 23. Admin guard semantics

`requireAdmin` must accept only these roles:

- `ADMIN`
- `SUPER_ADMIN`

No lower roles are accepted.

Do not rely on generic `admin.access` unless a later source explicitly requires that semantic.

## 24. Super-admin guard semantics

`requireSuperAdmin` must accept only `SUPER_ADMIN`.

There is no `SYSTEM` shortcut, wildcard, or implicit bypass.

## 25. Public guard semantics

`requirePublic` must use the narrowest source-authorized semantics.

Do not invent a redirect framework. Document ambiguity if the source is incomplete.

## 26. Onboarding guard treatment

`requireCompletedOnboarding` is deferred.

Do not implement a fake boolean, session metadata assumption, hardcoded `true`, route-derived completion, email-derived completion, or schema invention.

Point to deferred P1-005 work.

## 27. Active-user guard treatment

Do not create `requireActiveUser`.

Runtime active-user enforcement remains deferred.

A pure assertion over already trusted pre-resolved status may be documented separately, but it must not be confused with runtime user loading.

## 28. Route protection patterns

Implement the guard library and document server-side usage patterns.

Do not introduce actual route/layout wiring by default.

For `/app`, document an authentication-only pattern using `requireAuth`.

For `/admin`, document a future privileged pattern requiring trusted pre-resolved application roles. Do not pretend the current auth helper supplies roles.

For `/admin/settings`, document a future super-admin pattern. Do not add fake runtime wiring.

## 29. Server-authoritative enforcement

Privileged authorization must be server-authoritative.

Server mutations, privileged reads, and privileged route access must not rely on client-only checks.

## 30. Client-side limitations

Client-side checks are user-experience hints only.

Client state, client-provided roles, Supabase metadata roles, and arbitrary strings must not be trusted for privileged authorization.

## 31. Error behavior

Guard failures must fail closed.

Errors must distinguish unauthenticated, unauthorized, deferred, and invalid-input cases without leaking secrets or personally identifiable information.
