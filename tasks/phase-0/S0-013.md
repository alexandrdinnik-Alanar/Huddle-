## Task ID

`S0-013`

## Objective

Create a reusable, typed, schema-first boundary for validating future request bodies without implementing any real product API or form flow.

## Dependency

`S0-012` is merged. Validation failures must integrate with its standard API error envelope and `VALIDATION_ERROR` code.

## Source precedence

1. accepted ADRs and repository architecture rules;
2. actual merged repository implementation through S0-012;
3. Sprint 0 source for S0-013 where not superseded.

## Required dependency

Add Zod as an exact production dependency. This addition was explicitly approved by the user before implementation and must be recorded in the handoff.

## Required implementation

Create repository-equivalent files under `src/lib/validation/**`:

- `validate.ts` with `validateRequestBody(schema, body)`;
- `errors.ts` with `formatValidationError(error)`;
- a typed discriminated validation result;
- `schemas/example-profile.schema.ts`;
- `schemas/example-listing.schema.ts`;
- an index export surface.

Create:

- `docs/sprint-0/validation-pattern.md`;
- focused unit tests;
- `docs/execution/S0-013/HANDOFF.md`.

## Required behavior

- successful validation returns parsed schema output;
- failed validation returns an `ApiError` using code `VALIDATION_ERROR`, status 400, and the client-safe message `Invalid request body.`;
- formatted details separate field errors from form-level errors;
- error details contain schema messages and paths, never submitted values;
- schemas reject unknown keys by default;
- future routes can pass the validation error through `handleApiError` and preserve the S0-012 envelope.

## Example profile schema

Allow only:

- optional `name` string;
- optional `preferredLanguage` string.

The schema is illustrative only. It must not create a profile API or add sensitive family or child fields.

## Example listing schema

Allow only:

- `title`;
- `description`;
- optional `priceAmount`;
- `categoryId`;
- optional `approximateArea`;
- optional `city`;
- optional `country`.

It must not contain exact pickup address, home address, child name, child birthday, school, or other sensitive child data.

## Security and privacy

- treat the request body as `unknown`;
- never trust parsed client data for authorization;
- never include raw input values in validation errors;
- do not log request bodies or validation payloads;
- strict schemas must reject undeclared fields.

## Explicit exclusions

- real product APIs or forms;
- route handlers;
- database queries;
- Prisma schema or migrations;
- authorization changes;
- audit model/helper from S0-014;
- async validation or external lookups;
- localization framework for error messages.
