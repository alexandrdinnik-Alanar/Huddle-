## 9. Required seed scope

During implementation, do all of the following:

- create `prisma/seed.ts`
- use Prisma Client
- seed only initial Huddle roles
- use `upsert` keyed by `RoleKey`
- provide a human-readable name for every role
- provide a human-readable description for every role
- make the seed idempotent
- ensure repeated execution does not create duplicates
- do not delete existing roles
- do not revoke role assignments
- do not create users
- do not create `UserRole` assignments

## 10. Exact initial roles

Seed exactly these initial roles:

- `PARENT`
- `SELLER`
- `ORGANIZER`
- `EXPERT`
- `PARTNER`
- `MUNICIPALITY_USER`
- `MODERATOR`
- `ADMIN`
- `SUPER_ADMIN`
- `SYSTEM`

## 11. Idempotency requirements

- use `upsert`
- the unique key must be `Role.key` backed by the existing `RoleKey` enum
- running the seed repeatedly must not create duplicates
- repeated runs must preserve exactly one row per role key
- the seed must be deterministic
- do not use random values
- do not depend on timestamps for uniqueness

## 12. Upsert requirements

- upsert each role by its canonical `RoleKey`
- if a role already exists, update only the intended seed-managed metadata fields
- do not create alternate keys
- do not seed duplicate semantic roles under different keys
- do not delete rows as part of normal seed execution

## 13. Role metadata requirements

For every role:

- `key` must use the existing `RoleKey` enum
- `name` must be human-readable
- `description` must be concise and human-readable
- do not use placeholder text
- do not include secrets
- do not include credentials
- do not include user-specific data

## 14. Prisma Client requirements

- use the repository Prisma Client setup
- do not bypass Prisma with raw seed-only infrastructure unless strictly required
- `prisma/seed.ts` must be compatible with the repository Prisma configuration
- Prisma Client generation must succeed before handoff
- generated client artifacts in `node_modules` must not be committed
