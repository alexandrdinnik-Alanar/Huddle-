# S0-016 — Create canonical placeholder pages

## Task ID

`S0-016`

## Objective

Wire the existing canonical Sprint 0 placeholder route set to the base layouts from S0-015, reconcile placeholder metadata, and preserve strict feature-scope boundaries.

## Dependency

`S0-015` must be merged and its quality gates must pass.

## Repository reality

S0-002 already created 55 placeholder route definitions and explicit App Router entries. This task must reuse and reconcile that implementation rather than creating a second route registry or replacing it with product UI.

## Source precedence

1. `docs/02-architecture/canonical-route-map.csv`;
2. `src/lib/routes/placeholder-routes.ts` and merged route entrypoints;
3. `docs/01-product/screen-route-registry.csv` for design status;
4. layouts created in S0-015;
5. Sprint 0 Batch 6 source material where not superseded.

## Allowed files

- existing placeholder route entrypoints under `src/app/**`;
- `src/components/system/route-placeholder-from-path.tsx`;
- `src/lib/routes/placeholder-routes.ts` only for reconciled metadata corrections;
- focused route/placeholder tests under `tests/unit/**`;
- `docs/execution/S0-016/HANDOFF.md`.

## Requirements

1. Preserve all 55 unique placeholder routes unless a documented canonical conflict is proven.
2. Public routes use `PublicLayout`.
3. Auth routes use `AuthLayout`.
4. Authenticated app routes use `AppLayout`.
5. Admin routes use `AdminLayout`.
6. Each placeholder displays screen name, route, priority, surface, implementation status, and the statement that feature implementation is outside Sprint 0.
7. A route marked `UNDECIDED` in `screen-route-registry.csv` must not be styled as an approved production design.
8. A route marked `PRODUCTION` may reference its approved Figma coverage, but this task still implements only a placeholder.
9. Do not copy Claude Design HTML, inline styles, or PNG layouts into runtime code.
10. Do not introduce product data, API calls, Prisma queries, or feature state.

## Authorization boundary

- Client-side navigation visibility is not authorization.
- `/app` routes may use only the existing safe authentication boundary where the current architecture supports it.
- Do not invent app-user reconciliation, role loading, or onboarding completion.
- Do not wire admin authorization from untrusted provider metadata.
- Existing guard helpers and deferred boundaries must remain explicit in code and handoff.

## Acceptance criteria

- the existing 55-route placeholder manifest remains unique and test-covered;
- all route groups render through the correct base layout;
- placeholder content is consistent and explicitly non-production;
- no P1/P2/P3 feature implementation is introduced;
- dynamic route resolution remains covered;
- required quality gates pass;
- `docs/execution/S0-016/HANDOFF.md` is complete.

- `pnpm lint`

- `pnpm test`
