# S0-015 — Create base layouts

Status: `BLOCKED` until VIS-002 is approved and merged.

## Task ID

`S0-015`

## Objective

Create structural layout foundations for the public, auth, authenticated-app, and admin surfaces without implementing product features or inventing unapproved visual decisions.

## Dependencies

- `S0-014` merged.
- `VIS-002` completed, approved, documented, and merged.
- Canonical Figma layout frames for Public, Auth, App, and Admin exist with exact page names and node IDs recorded in the repository.

No approved Huddle product design currently exists in Figma. This task must not start until those dependencies are satisfied.

## Source precedence

1. accepted ADRs and repository security rules;
2. `docs/00-source-of-truth/design-source-of-truth.md`;
3. product-owner-approved Figma layout frames recorded by VIS-002;
4. locked brand system and existing design tokens;
5. Sprint 0 Batch 6 source material where not superseded.

PNG sets and the Claude Design corpus remain reference-only.

## Allowed files

- `src/components/layout/**`
- focused layout tests under `tests/unit/**`
- `docs/execution/S0-015/HANDOFF.md`
- task-specific documentation corrections required for the implementation

Do not change route pages, Prisma, APIs, auth identity reconciliation, or product modules.

## Required components

- `PublicLayout`
- `AuthLayout`
- `AppLayout`
- `AdminLayout`

Optional structural helpers:

- `PublicHeader`
- `PublicFooter`
- `AppNavigation`
- `AdminSidebar`
- `AdminHeader`
- `src/components/layout/index.ts`

## Requirements

1. Every layout accepts `children`.
2. Use existing React and Next.js App Router conventions.
3. Implement the exact approved VIS-002 layout structure without inventing visual decisions.
4. Use existing semantic CSS variables and approved brand components only.
5. Public and auth layouts may use `BrandLogo` where specified by Figma.
6. App and admin layouts must match their approved structural distinction without introducing product dashboards or feature controls.
7. Navigation content is structural placeholder content only.
8. No generated, redrawn, recolored, or alternate logo assets.
9. No raw Claude Design HTML or PNG may be copied into production components.
10. Any mismatch or missing Figma state must stop implementation and be returned to design rather than guessed in code.
11. Authorization remains server-side. This task must not fake role resolution or treat hidden navigation as protection.

## Security boundary

- Do not invent canonical app-user reconciliation.
- Do not infer trusted roles from Supabase provider identity.
- Do not add database role queries.
- Do not claim that a structural admin shell protects admin routes.
- Existing S0-011 guard helpers remain the authorization foundation; route wiring belongs only where a later task has trusted inputs.

## Acceptance criteria

- VIS-002 approval and node IDs are referenced in the handoff;
- all four layout components exist and compile;
- layouts match the approved structural frames;
- layouts use token-based styling;
- approved brand assets are used where specified;
- app and admin structures are visually and semantically distinct;
- no real marketplace, events, community, admin, finance, or municipality feature UI is introduced;
- focused tests pass;
- required quality gates pass;
- `docs/execution/S0-015/HANDOFF.md` is complete.
