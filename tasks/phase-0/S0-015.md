# S0-015 — Create base layouts

## Task ID

`S0-015`

## Objective

Create structural layout foundations for the public, auth, authenticated-app, and admin surfaces without implementing product features or inventing unapproved visual decisions.

## Dependencies

- `S0-014` merged.
- `VIS-001` merged.
- Canonical Figma source and route registry are authoritative for any material visual decision.

## Source precedence

1. accepted ADRs and repository security rules;
2. `docs/00-source-of-truth/design-source-of-truth.md`;
3. approved Figma master frames and components recorded in `screen-route-registry.csv`;
4. locked brand system and existing design tokens;
5. Sprint 0 Batch 6 source material where not superseded.

## Allowed files

- `src/components/layout/**`
- focused layout tests under `tests/unit/**`
- `docs/execution/S0-015/HANDOFF.md`
- task-specific documentation corrections required for the implementation

Do not change route pages, Prisma, APIs, auth identity reconciliation, or product modules.

## Required components

- `PublicLayout`
- `AuthLayout`
- `AppLayout`
- `AdminLayout`

Optional structural helpers:

- `PublicHeader`
- `PublicFooter`
- `AppNavigation`
- `AdminSidebar`
- `AdminHeader`
- `src/components/layout/index.ts`

## Requirements

1. Every layout accepts `children`.
2. Use existing React and Next.js App Router conventions.
3. Use existing semantic CSS variables and approved brand components only.
4. Public and auth layouts may use `BrandLogo` where appropriate.
5. App and admin layouts must be structurally distinct, but must not invent product dashboards or feature controls.
6. Navigation content is structural placeholder content only.
7. No generated, redrawn, recolored, or alternate logo assets.
8. No raw Claude Design HTML or PNG may be copied into production components.
9. A layout requiring a material visual decision not already approved in Figma must pause and record the missing Figma node instead of inventing the design.
10. Authorization remains server-side. This task must not fake role resolution or treat hidden navigation as protection.

## Security boundary

- Do not invent canonical app-user reconciliation.
- Do not infer trusted roles from Supabase provider identity.
- Do not add database role queries.
- Do not claim that a structural admin shell protects admin routes.
- Existing S0-011 guard helpers remain the authorization foundation; route wiring belongs only where a later task has trusted inputs.

## Acceptance criteria

- all four layout components exist and compile;
- layouts use token-based styling;
- approved brand assets are used where appropriate;
- app and admin structures are visibly and semantically distinct;
- no real marketplace, events, community, admin, finance, or municipality feature UI is introduced;
- focused tests pass;
- required quality gates pass;
- `docs/execution/S0-015/HANDOFF.md` is complete.

## Quality gates

- `python scripts/verify-brand-assets.py`
- `pnpm lint`
- `pnpm typecheck`
- `pnpm test`
- `pnpm build`
- `git diff --check`

## Stop condition

Stop after S0-015. Do not begin S0-016 in the same branch.
