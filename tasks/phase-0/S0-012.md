# S0-012 — Create API response and error helpers

## 1. Task ID

`S0-012`

## 2. Tracking title

`Create API response and error helpers`

## 3. Sprint 0 source title

`Create standard API response and error helpers`

The title difference is wording only and is not a scope conflict.

## 4. Objective

Create reusable, typed, safe API response, error, and pagination helpers before any real Huddle product APIs are implemented.

S0-012 is backend foundation only. It must establish a consistent API contract without implementing marketplace, event, report, admin, family, community, messaging, or other product endpoints.

`S0-011`

The dependency is satisfied once S0-011 is merged to `main`.

1. accepted ADRs and repository architecture rules
2. actual merged repository implementation through S0-011
3. roadmap for ordering, dependency, and tracking title
4. Sprint 0 source for detailed S0-012 scope where not superseded

## 7. Required source behavior

The standard API envelope must be consistent.

Success shape:

```json
{ "data": {}, "error": null }
```

List shape:

```json
{
  "data": {
    "items": [],
    "pagination": {
      "page": 1,
      "pageSize": 20,
      "total": 120,
      "hasMore": true
    }
  },
  "error": null
}
```

Error shape:

```json
{
  "data": null,
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "Invalid request body.",
    "details": {}
  }
}
```

## 9. Required helpers

Implement:

- `apiSuccess(data, init?)`
- `apiList(items, pagination, init?)`
- `apiError(code, message, details?, status?)`
- `handleApiError(error)`
- `createPagination({ page, pageSize, total })`

Exact TypeScript signatures may be narrowed during implementation, but must preserve the source behavior, type safety, stable envelope, and security requirements in this contract.

## 10. Required error codes

Define typed constants and/or a typed union covering exactly the source-authorized baseline:

- `UNAUTHORIZED`
- `FORBIDDEN`
- `NOT_FOUND`
- `VALIDATION_ERROR`
- `CONFLICT`
- `RATE_LIMITED`
- `RESOURCE_REMOVED`
- `RESOURCE_UNAVAILABLE`
- `OWNERSHIP_REQUIRED`
- `STATUS_NOT_ALLOWED`
- `SUSPENDED_USER`
- `MODERATION_REQUIRED`
- `SERVER_ERROR`

Do not invent product-specific error codes without explicit source authority.

## 11. HTTP status semantics

Use appropriate HTTP status codes.

The implementation must define a deterministic status policy for the required error codes. The mapping must be explicit, documented, and tested.

The implementation must not silently return HTTP 200 for known error conditions unless a specific repository standard explicitly requires it.

## 12. Response typing

Provide typed API response structures for:

- success envelopes
- error envelopes
- list envelopes
- pagination metadata
- typed error details

Prefer generic TypeScript types where useful.

The success and error branches must remain structurally unambiguous:

- success: `data` present, `error: null`
- error: `data: null`, `error` present

## 13. Pagination semantics

`createPagination({ page, pageSize, total })` must:

- return deterministic pagination metadata
- include `page`
- include `pageSize`
- include `total`
- include `hasMore`
- avoid division-by-zero behavior
- define behavior for invalid or non-finite numeric input
- fail safely rather than producing misleading metadata

The implementation must not add cursor pagination or product-specific pagination policy in S0-012.

## 14. Error normalization

`handleApiError(error)` must:

- avoid leaking raw internal errors
- avoid leaking stack traces
- avoid leaking secrets
- avoid leaking tokens
- avoid leaking database connection strings
- avoid leaking internal filesystem paths
- avoid leaking PII
- return a stable client-safe error envelope
- preserve explicitly recognized safe API errors
- convert unknown/unhandled errors to `SERVER_ERROR`

Unknown errors must fail closed.

## 15. Error details policy

`details` is optional.

Rules:

- details must be client-safe
- no raw thrown object serialization
- no raw stack
- no token/session dump
- no environment dump
- no Prisma/database error object exposure
- no Supabase error object exposure unless explicitly sanitized
- no arbitrary nested internal metadata by default

## 16. Interaction with S0-009 auth boundary

S0-012 must not change authentication architecture.

It must not:

- reconcile Supabase identities to application Users
- treat provider subject id as domain `User.id`
- load roles
- load active-user state
- load onboarding state

Authentication errors may be normalized to API error codes such as `UNAUTHORIZED`, but S0-012 must not invent missing identity architecture.

## 17. Interaction with S0-010 permissions

S0-012 must not change permission semantics.

It may normalize safe permission-denial outcomes to API error responses, but must not:

- load roles
- create DB permission models
- infer permissions from client input
- reinterpret the S0-010 role-permission map

## 18. Interaction with S0-011 guards

S0-012 must not change route-guard behavior.

It may normalize recognized safe guard denials into API error responses if implementation evidence justifies this, but must not:

- wire guards into product endpoints
- invent runtime role loading
- invent app-user resolution
- create route-wide authorization wiring

Any guard-error normalization must be explicit, deterministic, and tested.

## 19. Placeholder route policy

The Sprint 0 source permits one placeholder test/example route only if useful.

Default contract decision:

- no route file is required
- do not modify `src/app/**` by default
- prefer unit tests and documentation examples

A placeholder route may be added only after a factual blocker demonstrates clear value and the contract is explicitly corrected first.

## 20. Security requirements

- fail closed on unknown errors
- no raw stack traces to clients
- no secrets in responses
- no access tokens in responses
- no refresh tokens in responses
- no session dumps
- no PII leakage
- no internal filesystem paths
- no database URLs
- no raw Prisma error leakage
- no raw Supabase error leakage
- no arbitrary object serialization for unknown errors
- stable deterministic error codes
- server errors use safe generic client messages
